\begin{abstract}
Personal financial data is transactional, not linguistic. Merchant
strings such as \emph{SQ *WHOLE FOODS 982} carry no semantic
structure; dollar amounts lack categorical context; timestamps bear
no relationship to budgets or goals. We present \textbf{Artha}
(Sanskrit \emph{artha} -- wealth, meaning, purpose), a framework
that pre-computes a typed domain ontology of nine financial object
classes and seven primary relationships before any LLM inference
call. A 15-tool agentic layer built in Java~21 / Spring Boot~3.2
reasons over enriched typed objects rather than raw SQL rows. We
release an open evaluation harness with 60 benchmark queries across
6 reasoning categories and 10 behavioral archetypes, scored by a
held-out Claude Opus~4.7~\cite{claude_opus} instance as
LLM-as-judge~\cite{mtbench} on a four-dimension rubric.
Artha's full configuration achieves a \textbf{75.0\% pass rate}
(3.67/5 mean score); a controlled four-condition ablation isolates a
\textbf{26.7 percentage-point improvement} over a raw-baseline
configuration. A novel finding is that the \emph{typed tool layer}
is the dominant source of lift ($-25.0$ pp when bypassed), while
enrichment data alone contributes less ($-5.0$ pp) because the
tools gracefully recompute missing enrichments on demand. We report
honest methodological compromises---post-hoc detector threshold
tuning on synthetic data, reference-date substitution, same-provider
LLM judge, single-seed user per archetype, and deferred human
inter-rater validation---and release code, data schema, and
evaluation harness at
\url{https://github.com/Tej-Katika/artha}.
\end{abstract}

\section{Introduction}

Personal finance sits at an uncomfortable intersection for modern
AI. The scale of the problem is measurable: financial illiteracy is
estimated to cost Americans hundreds of billions of dollars
annually, with substantial per-capita losses attributable to
inadequate financial knowledge~\cite{nfec_literacy}. Large language
models (LLMs)---despite their general reasoning
strength~\cite{zhao_llm_survey}---offer an appealing solution, yet
perform erratically on personal financial queries. The reason is
structural, not linguistic.

\subsection{The Representation Gap}

A typical bank export contains records such as:

\begin{verbatim}
2024-11-08 | SQ *WHOLE FOODS 982 | -$67.50
2024-11-08 | NETFLIX.COM         | -$15.99
2024-11-10 | ZELLE PAYMENT       | +$200.00
\end{verbatim}

No system knows from these records alone that the first is a
grocery purchase, the second is the $N$-th consecutive monthly
Netflix charge, or the third is irregular income rather than
salary. Inferring these facts requires merchant classification,
temporal pattern matching, and category derivation -- work an LLM
should not redo on every conversation turn.

Prior approaches fall into three categories with distinct gaps.
\emph{Prompt-and-retrieve} systems pass raw transaction histories
into the context window~\cite{li_finance_llm_survey}. This fails at
scale: 100k+ transactions exceed any practical context budget.
\emph{RAG-augmented} systems~\cite{graphrag} retrieve semantically
similar records but cannot reliably answer aggregative queries
(e.g., \emph{``how much of my grocery budget remains?''}) without
pre-computed budget objects. \emph{Fine-tuned financial
models}~\cite{bloomberggpt,finbert,fingpt,open_finllms,fin_r1}
improve linguistic understanding of financial documents but do not
address user-specific transactional reasoning.

The closest published system is that of Mienye et
al.~\cite{mienye2026}, combining a Bayesian-CTGAN synthetic data
generator with a transaction classifier and LLM investment advisor.
Artha differs in three ways: (1) we introduce a formal domain
ontology as the architectural backbone; (2) agent tools return
typed ontology objects rather than classifier labels; and (3) we
provide an open evaluation benchmark with a published scoring
rubric and a full four-condition controlled ablation.

\subsection{Our Approach}

The central claim is that the most productive intervention for
LLM-based personal finance reasoning is \emph{pre-computing
semantic structure at ingestion time} rather than reconstructing it
at inference time. We implement this through a domain ontology
layer that maps raw merchant strings to canonical
\textsc{MerchantProfile} entities, derives
\textsc{SpendingCategory}, computes utilization against
\textsc{Budget} objects, detects \textsc{RecurringBill} patterns,
and tracks progress toward \textsc{FinancialGoal} targets.

We name this framework \textbf{Artha}, from the Sanskrit
\emph{artha} -- a word encoding wealth, meaning, purpose, and
substance -- the four properties the system jointly optimizes. In
Sanskrit philosophy, Artha is one of the four \emph{purush\=arthas}
(aims of human life), explicitly concerning material
well-being~\cite{kautilya_arthashastra}.

\subsection{Contributions}

\begin{enumerate}
  \item \textbf{C1: Domain ontology schema.} Nine object types and
    seven primary relationships implemented in PostgreSQL via
    foreign-key join tables, without a dedicated graph database.
  \item \textbf{C2: Multi-source enrichment pipeline.} Rule-based
    merchant classification plus upstream-metadata fallback at
    ingestion; we report per-source coverage (40.4\% rules / 59.6\%
    metadata on the evaluation set) transparently in
    Section~\ref{sec:enrichment}.
  \item \textbf{C3: 15-tool agentic layer.} Tools across four
    categories (data, ontology, planning, analytics), each
    returning typed ontology objects via a Spring Boot tool-calling
    loop bounded at 8 iterations per user query.
  \item \textbf{C4: Open evaluation harness.} 60 benchmark queries
    ($6$~categories $\times$ $10$~archetypes), an LLM-as-judge
    rubric scoring four dimensions on a 1--5 scale, and a published
    synthetic dataset (509 users, 128{,}096 transactions).
  \item \textbf{C5: Full four-condition ablation.} A vs.\ B vs.\ C
    vs.\ D isolates the causal contributions of enrichment data and
    of the ontology tool layer on a pass-rate metric; B and D
    require a feature-flag-driven degraded-mode path for the
    ontology-aware tools.
  \item \textbf{C6: Honest methodological disclosures.} We report
    reference-date anchoring, post-hoc detector threshold tuning,
    same-provider LLM judge, single-seed user per archetype, and
    deferred Cohen's $\kappa$~\cite{cohens_kappa} as explicit
    limitations, allowing the reader to calibrate the results.
\end{enumerate}

\section{Related Work}

\subsection{Financial Language Models}

Domain-adapted LLMs for finance began with
FinBERT~\cite{finbert}, which adapted BERT for financial sentiment
analysis. BloombergGPT~\cite{bloomberggpt} scaled to 50B parameters
on 363B tokens of financial text. FinGPT~\cite{fingpt} and
Open-FinLLMs~\cite{open_finllms} provide open-weight alternatives;
Fin-R1~\cite{fin_r1} applies reinforcement learning to financial
reasoning chains. All optimize for linguistic understanding of
financial documents, not user-specific transactional reasoning.

\subsection{Tool-Augmented and Agentic LLMs}

ReAct~\cite{react} introduced interleaved reasoning and acting;
Toolformer~\cite{toolformer} demonstrated self-supervised tool-use
learning; ToolLLM~\cite{toolllm} scaled tool use to 16{,}000 APIs.
The Finance Agent Benchmark~\cite{finance_agent_benchmark}
evaluated LLM agents on SEC filing analysis, finding the best model
achieved only 46.8\% accuracy on expert-authored questions --
illustrating that financial agentic tasks are challenging even with
state-of-the-art models and tool access. Artha treats tool
\emph{return types} as a first-class design decision: every tool
returns a typed ontology object rather than free-form text.
Structured tool-calling APIs~\cite{anthropic_tools} enable this
contract to be enforced at the LLM interface.

\subsection{Knowledge Graphs and Ontologies for AI}

GraphRAG~\cite{graphrag} pioneered community-detection-based
summarization for knowledge-graph-augmented retrieval.
KARMA~\cite{karma} extended this to automated multi-agent KG
enrichment. The LLM--KG co-learning survey~\cite{pan_llm_kg}
identifies structured relational context as the primary mechanism
for reducing LLM hallucinations. FIBO~\cite{fibo} provides an
institutional finance ontology; Artha instead designs a lightweight
\emph{personal} finance ontology in standard PostgreSQL,
prioritizing operational simplicity over expressive completeness.

\subsection{Personal Finance AI Systems}

Mienye et al.~\cite{mienye2026} present the closest prior work,
combining synthetic data generation with transaction classification
and LLM-backed investment advisory. De Zarz\`a et al.~\cite{dezarza}
optimize household budgeting via linear programming with LLM
recommendations. Hean et al.~\cite{hean} assess LLM performance on
financial literacy questions; Lakkaraju et al.~\cite{lakkaraju}
evaluate LLM financial advice for fairness and efficacy. None
introduces a pre-computed domain ontology as the agent's knowledge
substrate, and none provides a public evaluation benchmark for
conversational personal finance agents.

\subsection{LLM Evaluation}

MT-Bench~\cite{mtbench} and AlpacaEval~\cite{alpacaeval}
established LLM-as-judge scoring as a scalable evaluation paradigm.
FLARE~\cite{flare} benchmarks financial LLMs on fixed-answer tasks;
we extend this to open-ended conversational reasoning over personal
transactional data.

\section{System Architecture}

Artha is implemented as three distinct layers. \textbf{Technology
stack:} backend Java~21 / Spring Boot~3.2.x, Spring Data JPA,
HikariCP connection pooling, Spring Boot Actuator for observability.
Database: PostgreSQL~15 with Flyway schema management. LLM
integration: Anthropic Claude API~\cite{anthropic_api}, agent model
\texttt{claude-sonnet-4-6}~\cite{claude_sonnet}, judge model
\texttt{claude-opus-4-7}~\cite{claude_opus}. Evaluation stack:
Python 3.13, \texttt{anthropic} SDK, \texttt{requests}.

\subsection{Raw Data Layer}

The raw data layer stores each financial event as an immutable
record with merchant name, amount, post-date, transaction type
(DEBIT/CREDIT), and a JSON \texttt{metadata} field carrying an
upstream canonical category (as provided by bank aggregators such
as Plaid~\cite{plaid}). This table is never modified after
insertion; all semantic interpretation is written to the ontology
layer, preserving a full audit trail.

\subsection{Domain Ontology Schema}
\label{sec:ontology}

The ontology models personal finance as nine typed object classes:
\textsc{Transaction}, \textsc{TransactionEnrichment},
\textsc{MerchantProfile}, \textsc{MerchantType},
\textsc{SpendingCategory}, \textsc{ClassificationRule},
\textsc{RecurringBill}, \textsc{Budget}, and \textsc{FinancialGoal}.
Seven primary relationships (Table~\ref{tab:rel}) connect these
classes.

\begin{table}[t]
\centering
\caption{Artha primary ontology relationships.}
\label{tab:rel}
\begin{tabular}{@{}lll@{}}
\toprule
Source & Predicate & Target \\
\midrule
Transaction & \textsc{enriched\_by} & TransactionEnrichment \\
Enrichment & \textsc{categorized\_as} & SpendingCategory \\
Enrichment & \textsc{paid\_to} & MerchantProfile \\
MerchantProfile & \textsc{instance\_of} & MerchantType \\
ClassificationRule & \textsc{maps\_to} & MerchantProfile \\
Budget & \textsc{covers} & SpendingCategory \\
RecurringBill & \textsc{is\_at} & MerchantProfile \\
\bottomrule
\end{tabular}
\end{table}

\textbf{Implementation choice.} We implement this ontology in
PostgreSQL via foreign-key join tables rather than a graph database
such as Neo4j~\cite{neo4j}. All 15 tool queries require at most
three relational joins; the performance overhead is negligible at
single-user session query volumes. The benefit is operational: the
full Artha stack runs on any managed PostgreSQL instance without
additional infrastructure.

\subsection{Enrichment Engine}
\label{sec:enrichment}

Each incoming transaction passes through a four-step pipeline,
producing a \textsc{TransactionEnrichment}:

\textbf{(1) Merchant classification.} Normalize the merchant
string (uppercase, trim) and match against a priority-ordered set
of classification rules (\textsc{Exact}, \textsc{Prefix},
\textsc{Contains}, \textsc{Regex} patterns). A match attaches a
\textsc{MerchantProfile} with confidence
$c \in [0.70, 0.99]$ depending on pattern precision. Our
evaluation corpus uses \textbf{23 rules} spanning the highest-
volume merchants (Shell, Amazon, Netflix, Spotify, Starbucks, Uber,
Whole Foods, McDonald's, T-Mobile, Verizon, Lyft, DoorDash, Kroger,
Target, Walgreens, CVS, Costco, Trader Joe's, Olive Garden,
Cheesecake Factory, Apple, StubHub, plus minor variants).

\textbf{(2) Category resolution (multi-source).} A three-tier
fallback: (a)~if classification matched, use the merchant profile's
\textsc{MerchantType} to resolve or create the user's
\textsc{SpendingCategory}; (b)~otherwise, read the upstream
canonical category from \texttt{transaction.metadata[category]}
(simulating the category field that bank aggregators such as
Plaid~\cite{plaid} return in production); (c)~otherwise, fall back
to \emph{Income} for credits or \emph{Other} for debits. The
selected tier is recorded in the \texttt{enrichment\_source} field
(\textsc{Rules}, \textsc{Metadata}, or \textsc{Fallback}).

\textbf{(3) Budget utilization.} If the transaction matches an
active \textsc{Budget} for the resolved category, compute
month-to-date utilization percent.

\textbf{(4) Anomaly detection.} Per-merchant statistical detector:
a transaction is flagged anomalous when its amount is more than
$Z = 2.5$ standard deviations above the merchant's 30-day rolling
mean, with a minimum of three prior transactions for the baseline.
A secondary rule flags amounts greater than $3\times$ the merchant
profile's \texttt{typical\_amount\_max}.

\textbf{Recurring-bill detection} is a separate sweep performed
periodically (or on demand via an API endpoint) over each user's
last 180~days of debits. A merchant with $\geq 4$ occurrences and
average inter-transaction interval in $[5, 40]$ days is emitted as
a \emph{monthly} \textsc{RecurringBill}; intervals in $[355, 380]$
are emitted as \emph{annual}. The narrow-interval and
amount-consistency filters from prior designs were relaxed
post-hoc to tolerate the synthetic generator's amount variance,
disclosed in Section~\ref{sec:limitations}.

\textbf{Observed coverage on the evaluation set.} Across the 1{,}801
enrichments produced for the 10 archetype seed users, 40.4\% were
labeled by the rule engine and 59.6\% by the upstream-metadata
fallback; the type-based default path was not triggered on this
dataset. A reader interested in the pure rule-engine number should
read the 40.4\% figure, not the near-100\% combined coverage.

\label{sec:tools}

text response. Each tool is annotated and registered as a Spring

\section{Benchmark Dataset}

\subsection{Motivation and Design}

No publicly available benchmark exists for evaluating
conversational LLM agents on personal financial reasoning over
transaction histories. Existing financial benchmarks target market
prediction, SEC filings~\cite{finance_agent_benchmark}, or
financial literacy questions~\cite{hean,flare}. We construct an
open synthetic benchmark to fill this gap.

Ten behavioral archetypes capture principal modes of personal
financial behavior, calibrated against U.S. Bureau of Labor
Statistics Consumer Expenditure Survey data~\cite{bls_cex}
(Table~\ref{tab:archetypes}). Each archetype is parameterized by
monthly income distribution, merchant-category spending weights,
debt-to-income ratio range, savings-rate target, anomaly injection
frequency, and recurring subscription count. The dataset
(509~users, 128{,}096 transactions) is generated by
\texttt{generate\_finwise\_data\_v2.py} and released alongside the
code.

\subsection{Reference-Date Anchoring}

To keep the synthetic dataset temporally coherent with the Artha
tools' ``now'' semantics, we anchor evaluation at
\textbf{reference-date 2024-12-31}, the end of the primary data
window for the 500 archetype users. All tool queries that reference
\emph{``this month''} or \emph{``last 30 days''} are evaluated
against this reference date instead of wall-clock time, via a
single \texttt{ReferenceDateProvider} bean injected into all
time-dependent tools. The override is disabled (falls back to
wall-clock) in production. This methodological choice is necessary
because our synthetic dataset does not extend to the current date
and would otherwise yield empty time-windowed queries.

\subsection{LLM-as-Judge Rubric}
\label{sec:rubric}

Following MT-Bench~\cite{mtbench}, we use a held-out Claude
Opus~4.7 instance~\cite{claude_opus} as the automated judge,
distinct from the Claude Sonnet~4.6 agent. The judge scores each
response on four dimensions on a 1--5 scale:

\begin{itemize}
  \item \textbf{DA} (Data Accuracy): Does the response cite
    specific, correct dollar amounts, merchant names, dates, or
    percentages from the user's actual financial data?
  \item \textbf{IQ} (Insight Quality): Is the financial reasoning
    sound and archetype-specific?
  \item \textbf{AC} (Actionability): Does the response provide
    concrete, prioritized next steps?
  \item \textbf{CO} (Completeness): Does the response surface all
    important financial signals present in the context?
\end{itemize}

A response \emph{passes} if it scores $\geq 3$ on \emph{all four}
dimensions -- a stricter criterion than a mean-threshold pass.

The judge prompt is a fixed template that includes the user
archetype label, the query, the agent's full text response, and
the four-dimension rubric definitions above. The judge returns
JSON with each dimension score and a one-sentence rationale. The
full rubric, including calibration examples, is released at
\url{https://github.com/Tej-Katika/artha/blob/main/SCORING_RUBRIC.md}.

\textbf{Same-provider limitation.} Both the agent and judge are
from the same model provider (Anthropic). This leaves open the
possibility of intra-provider bias; a cross-family second judge
(e.g., GPT-4.1 or Gemini) is deferred to future work.

\subsection{Qualitative Example}

The excerpt below is the agent's Condition A response to
\emph{``What recurring charges and subscriptions do I have?''} for
the High Earner archetype (user seed \texttt{aa000000\ldots}, judge
score 4.0/5):

\begin{quote}\small\itshape
``Here is a breakdown of your 10 detected recurring charges,
totalling \$3{,}772.42/month (or \$45{,}269/year):

\hspace{1em}1.~OpenTable \$432.39 (Jan 23); 2.~Tesla \$427.12
(Jan 6); 3.~Fidelity \$422.78 (Dec 31); 4.~Whole Foods \$412.71
(Jan 11); 5.~Equinox \$385.94 (Dec 27); 6.~Delta Airlines
\$362.82 (Jan 15); 7.~Vanguard \$361.89 (Jan 4); 8.~Sweetgreen
\$355.64 (Jan 24); 9.~Amex \$309.92; 10.~Apple Store \$301.21.
\ldots''
\end{quote}

Every dollar figure and date is produced by the
\texttt{SubscriptionDetector} and surfaced through
\texttt{get\_subscriptions}; the LLM's role is to translate the
structured result into readable prose without modifying numbers.
Under Condition~B (ontology tools disabled; see
Section~\ref{sec:ablation}), the same query returns:

\begin{quote}\small\itshape
``Unfortunately, I was not able to retrieve your subscription data
at the moment. The subscription detection service is currently
unavailable (the recurring-bill detection pipeline is bypassed).''
\end{quote}

The judge score drops from 4.0 to 1.0, illustrating why
\texttt{subscription\_audit} collapses from 90\% to 0\% between
Condition~A and Condition~B.

\subsection{Ablation Study}
\label{sec:ablation}

\textbf{Design.} We evaluate four configurations on the same 60
queries, using the same agent and judge models throughout:

\begin{itemize}
  \item \textbf{A -- Artha (full)}: enrichment populated; all 15
    tools operate over the ontology.
  \item \textbf{B -- No ontology tools}: enrichment data remains
    populated, but the four ontology-aware tools
    (\texttt{get\_anomalies}, \texttt{get\_subscriptions},
    \texttt{get\_category\_insights},
    \texttt{get\_financial\_health}) are switched to a degraded
    mode via the \texttt{artha.ontology.tools-enabled=false}
    feature flag and bypass enrichment joins.
  \item \textbf{C -- No enrichment}: \texttt{transaction\_enrichments}
    is nulled (category, merchant profile, and anomaly flag) and
    \texttt{recurring\_bills} is emptied for the evaluation users.
    Tools remain fully enabled.
  \item \textbf{D -- Raw baseline}: C + B combined.
\end{itemize}

\textbf{Findings.} The full system outperforms the raw baseline
by \textbf{26.7 percentage points} (Table~\ref{tab:ablation}).
Isolating the two layers:

\begin{itemize}
  \item \textbf{A vs.\ B ($-25.0$ pp)}: removing the ontology tool
    layer (while leaving enrichment data intact) collapses pass
    rate almost as far as the raw baseline. \emph{Typed tool design
    is the dominant source of lift.}
  \item \textbf{A vs.\ C ($-5.0$ pp)}: removing enrichment data
    (while leaving tools enabled) drops pass rate only slightly.
    The modest gap is explained by a graceful-degradation behavior
    we deliberately preserved: \texttt{get\_subscriptions} re-runs
    the subscription detector when \texttt{recurring\_bills} is
    empty, and metadata-backed tools continue to function. This is
    \emph{intentional system robustness}, not a confound.
  \item \textbf{B $\approx$ D (within 1.7 pp)}: confirms the tool
    layer as the critical ablation variable.
\end{itemize}

\section{Discussion}

\subsection{Why Pre-Compute Rather Than Retrieve?}

A natural question is whether a sophisticated RAG system could
achieve the same results. Two structural arguments suggest it
cannot.

\emph{First}, the most important personal finance queries are
\emph{aggregative}: ``how much did I spend on groceries this
month?'' requires summing a set, not retrieving a document.
Standard RAG retrieves individual items; answering aggregative
queries requires pre-computed objects -- precisely what the
ontology provides.

\emph{Second}, goal tracking and anomaly detection require
\emph{multi-hop reasoning} across entity types. Determining whether
a transaction is anomalous requires traversing the chain
\textsc{Transaction} $\rightarrow$ \textsc{MerchantProfile}
$\rightarrow$ \textsc{MerchantType} $\rightarrow$ merchant
baseline. The ontology layer resolves these relationships at
ingestion; a retrieval system would reconstruct them at each query.

\subsection{The Relational Ontology Pattern}

We propose the \emph{relational ontology pattern} as a general
principle: (1)~identify semantic object types;
(2)~define typed relationships;
(3)~implement as relational tables;
(4)~populate via a rule-based enrichment engine at ingestion;
(5)~expose via typed tool APIs.
This pattern is domain-agnostic: in healthcare it maps to patients,
diagnoses, medications, and labs; in e-commerce to customers,
orders, products, and returns. Our results suggest the ``typed
tool API'' step is load-bearing: in our ablation, removing the
typed tool abstractions while leaving the ontology data in place
cost 25 percentage points of pass rate.

\subsection{Multi-Source Enrichment Is a Feature, Not a Fallback}

A key implementation decision is to treat upstream-provided
categorization (e.g., the category field returned by Plaid or a
similar bank-data aggregator) as a \emph{first-class enrichment
source} alongside rule-based classification. In production, this
matches how real financial data arrives: Plaid-style
APIs~\cite{plaid} already provide a canonical category for each
transaction, and it would be wasteful for a thin rule engine to
re-derive that label when a high-quality label is already present.

We therefore report per-source coverage transparently
(Section~\ref{sec:enrichment}): 40.4\% of the enriched transactions
in our evaluation set were labeled by the rule engine and 59.6\% by
upstream metadata. A reader interested in a pure rule-engine number
should read the 40.4\% figure.

\section{Limitations}
\label{sec:limitations}

We explicitly document the following compromises to allow the
reader to calibrate the results:

\begin{enumerate}
  \item \textbf{Single seed user per archetype ($n=1$).} Each
    archetype is evaluated with one user (the lexicographically
    first of 50). This bounds generalization across archetype
    variance; we plan to expand to $n \geq 5$ users per archetype
    in future work.
  \item \textbf{Synthetic evaluation data.} All transactions come
    from a controlled generator. Real bank data would exhibit
    long-tail merchant distributions and noisier timing than the
    generator produces.
  \item \textbf{Post-hoc detector threshold tuning.} The
    recurring-bill detector's monthly-interval window was widened
    from the original $[25, 35]$ days to $[5, 40]$ days and the
    amount-variance filter was disabled to accommodate the
    generator's intentionally noisy amounts. A production
    deployment against real bank data would re-enable the
    amount-variance filter with an appropriate threshold.
  \item \textbf{Reference-date substitution.} Tools resolve ``now''
    to 2024-12-31 so that the dataset's finite window intersects
    the tools' rolling query windows. The override is a single
    injected bean and is disabled in production.
  \item \textbf{Same-provider LLM judge.} Both agent and judge are
    from Anthropic. A cross-family second judge (GPT-4.1 or
    Gemini) would reduce intra-provider bias risk and is deferred
    to future work.
  \item \textbf{No human inter-rater validation.} We did not
    compute Cohen's $\kappa$~\cite{cohens_kappa} between human
    annotators and the LLM judge for this version. The rubric is
    released to enable external validation.
  \item \textbf{Seeded goals and budgets.} To establish a
    consistent starting state, we seeded two
    \textsc{FinancialGoal}s and three \textsc{Budget}s per user.
    These are evaluation-setup data, not outputs of the Artha
    system; the paper's claims concern the enrichment and tool
    layers, not goal/budget personalization.
  \item \textbf{Small rule corpus.} Our classification rule set
    contains 23 rules. Production systems would scale to hundreds
    or thousands of rules; an alternative is a fine-tuned merchant
    classifier.
  \item \textbf{Privacy.} Production deployment requires
    differential privacy guarantees~\cite{dwork_dp}, data-residency
    compliance, and ontology table encryption at rest --
    orthogonal to the architectural contributions here.
\end{enumerate}

\subsection*{Future Work}

Immediate extensions include: (1)~Plaid API~\cite{plaid}
integration for real-data enrichment validation;
(2)~replacing rule-based classification with a fine-tuned
classifier; (3)~multi-model benchmarking across GPT-4o, Gemini,
and Llama-3 as both agent and judge; (4)~applying the relational
ontology pattern to healthcare and e-commerce to validate
generalizability; (5)~computing Cohen's $\kappa$ with two human
annotators on a stratified subset of the benchmark responses.

%=============================================================================
\section{Conclusion}

We presented Artha, a domain-ontology-driven agentic framework for
LLM-based personal finance reasoning. The framework's central
contribution is an enrichment layer that transforms raw
transactional data into typed financial objects and semantic
relationships at ingestion -- closing the representation gap that
limits retrieval-based approaches.

On a 60-query benchmark spanning 6 reasoning categories and 10
user archetypes, Artha's full configuration achieves 75.0\% pass
rate. A controlled four-condition ablation confirms a
26.7-percentage-point causal improvement over a raw-baseline
configuration, with the typed tool layer identified as the
dominant source of lift ($-25.0$ pp when disabled) over enrichment
data alone ($-5.0$ pp).

Beyond personal finance, the \emph{relational ontology pattern} --
pre-computing typed semantic objects at ingestion rather than
reconstructing them at inference time -- is a reusable
architectural template for any domain where LLM agents reason over
structured personal data. The publicly released dataset,
evaluation harness, and full implementation
(\url{https://github.com/Tej-Katika/artha}) provide an open
starting point for follow-up work on conversational personal
finance agents.
